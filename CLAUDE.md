# GoodMathScore — карта проекта

Мобильный тренажёр устного счёта на чистом HTML/CSS/JS. Развёртывается как Telegram Mini App (но работает и в обычном браузере). Сейчас всё в трёх файлах: [index.html](index.html), [main.js](main.js), [style.css](style.css) + темы в [thems/](thems/). Проект подлежит реформированию — этот документ фиксирует ту логику, которую нужно сохранить или сознательно поменять при рефакторинге.

## Среда выполнения

- **Целевая платформа — мобильный экран в Telegram WebApp.** Поэтому используется `window.Telegram.WebApp.expand()` + `disableVerticalSwipes()` на старте, а размеры в CSS почти везде в `vh`/`vw`. Десктоп — не приоритет.
- **Определение, что мы внутри Telegram:** `navigator.userAgent.includes("Telegram")` (helper `inTelegram()` в [main.js](main.js)). От этого зависит, дублируется ли статистика в Telegram CloudStorage. Вне Telegram пишется только в localStorage — приложение в браузере полностью функционально, статистика не теряется.
- **Внешние зависимости** подключены через CDN: jQuery, Raphael, Morris.js (для графиков), Telegram WebApp JS, Google Material Icons. Логика ползунков написана на jQuery, остальное — vanilla JS. При рефакторинге это главное решение: оставлять jQuery/Morris или переписывать.

## Три экрана-страницы (Swiper-слайды в одном HTML)

Три экрана живут как три `.swiper-slide` внутри общего `.main-swiper` (Swiper.js v11, подключён через CDN). Палец-свайп **отключён** (`allowTouchMove: false`, `simulateTouch: false`) — переход возможен только программно через `mainSwiper.slideTo(index)`.

Порядок слайдов (константы в верхней части [main.js](main.js)):

| index | константа | id | назначение |
|---|---|---|---|
| 0 | `SLIDE_STATISTIC` | `#statistic` | Графики Morris.Line: примеры/время/ошибки по дням месяца. |
| 1 | `SLIDE_MAIN1` | `#main1` | Главный экран настроек. Чекбоксы операций, два диапазона, ползунок количества, пресеты «Малый»/«Большой», «Начать». **`initialSlide` — этот.** |
| 2 | `SLIDE_MAIN2` | `#main2` | Экран решения: домой/таймер/счёт, пример, область ответа, цифровая клавиатура. Физическая клавиатура не подключена. |

Сам Swiper инициализируется в `DOMContentLoaded` и хранится в глобальной `mainSwiper`. Все функции-переходы (`statisticOpen`, `statisticClose`, `fromHomeToExample`, `fromExampleToHome`) теперь дёргают `mainSwiper.slideTo(...)` вместо `classList.add/remove('none')`.

CSS-класс `.none` (`display:none`) **остался** — он всё ещё используется для скрытия `#win-message` внутри `#main1`. Не путать с переключением экранов.

Плюс выезжающая панель тем `#different-theme-block` — она вне Swiper-контейнера, оверлеит всё через `position: absolute`.

## Хранение настроек (зеркало localStorage ↔ CloudStorage)

Все пользовательские настройки живут в `localStorage` (источник синхронных чтений UI), но каждая запись зеркалится в Telegram CloudStorage через `cloudSet(key, value)` — это даёт переносимость между устройствами/переустановками. На старте `bootstrapCloudStorage()` подтягивает значения из облака → `localStorage` через `CloudStorage.getItems(...)`, после чего `applyUIFromLocalStorage()` переприменяет настройки.

**Зеркалируемые ключи** (`CLOUD_SETTINGS_KEYS` в [main.js](main.js)):

| ключ | формат | пишет |
|---|---|---|
| `values` | CSV (см. ниже) | `saveCurrentValues`, `applyRangesBounds` (clamp) |
| `userTheme` | имя темы | `applyTheme` |
| `rangesBounds` | JSON | `saveRangesBounds` |
| `rangesPresets` | JSON | `saveRangesPresets` |
| `sizeSettings` | JSON | `saveSizeSettings` |
| `activePreset` | `'big' \| 'small' \| ''` | `setActivePreset` |

`stats_v2` идёт по отдельному каналу (`_persistStats` пишет cloud+local напрямую) с другой логикой загрузки — слияние max-per-field, не «cloud wins», см. ниже.

**Bootstrap-порядок в DOMContentLoaded:**
1. Swiper init, `Telegram.WebApp.expand()`.
2. `applyUIFromLocalStorage()` — мгновенный первый кадр с тем, что сейчас в localStorage.
3. `statsLoad()` — async, прогрев кэша статистики (merge cloud+local).
4. `bootstrapCloudStorage().then(applyUIFromLocalStorage)` — переприменение если в облаке более свежие настройки.

Вне Telegram (`!inTelegram()`) `cloudSet` работает как обычный `localStorage.setItem`, `bootstrapCloudStorage` сразу резолвится — приложение полностью функционально, просто без переноса между устройствами.

## Контракт массива `values` (главная структура данных настроек)

Массив `values` хранит выбор пользователя на главном экране. Индексы строго фиксированы — см. комментарий в [main.js:2](main.js#L2):

```
0  +  (чекбокс сложения,        bool как строка "true"/"false")
1  −  (чекбокс вычитания)
2  ⋅  (чекбокс умножения)
3  ∶  (чекбокс деления)
4  ⏱ (чекбокс таймера — показывать ли время на экране решения)
5  min для +/−   (нижняя граница операндов сложения и вычитания)
6  max для +/−
7  min для ⋅/∶  (нижняя граница операндов умножения и деления)
8  max для ⋅/∶
9  количество примеров в раунде
```

`values` сериализуется в `localStorage` через **`cloudSet('values', values)`** — массив неявно превращается в CSV-строку (`Array.prototype.toString`), а при чтении парсится через `.split(',')`. Поэтому булевы значения там лежат как строки `"true"`/`"false"`, и проверки сделаны через `== "true"`. Это важная хрупкая деталь — при рефакторинге лучше переходить на JSON.

## Хранение настроек (`cloudSet` → localStorage + CloudStorage)

Все пользовательские настройки пишутся через единую обёртку `cloudSet(key, value)`. Она всегда пишет в `localStorage` (sync, источник истины для UI-чтения) и дополнительно в `Telegram.WebApp.CloudStorage` (async, fire-and-forget) — но только если `_cloudWriteEnabled === true`, что выставляется в конце `DOMContentLoaded`-обработчика. Это нужно, чтобы CloudStorage-вызовы при parse-time top-level коде (`applyRangesBounds`, `applySizeSettings`, top-level бутстрап) не конкурировали с инициализацией Swiper и Telegram WebApp — прошлая попытка mirror-on-startup ломала навигацию в Telegram именно из-за такой гонки.

**Зеркалируемые ключи** (все через `cloudSet`):

| ключ | формат | пишет |
|---|---|---|
| `values` | CSV (см. выше) | `saveCurrentValues`, `applyRangesBounds` (clamp) |
| `userTheme` | имя темы | `applyTheme` |
| `rangesBounds` | JSON | `saveRangesBounds` |
| `rangesPresets` | JSON | `saveRangesPresets` |
| `sizeSettings` | JSON | `saveSizeSettings` |
| `activePreset` | `'big' \| 'small'` или удалён | `setActivePreset` |

**Что НЕ зеркалится сейчас**:
- Чтение настроек из CloudStorage на старте (cross-device sync) — намеренно не реализовано, чтобы не повторить поломку Telegram-инициализации. Если понадобится — отдельная задача.
- Статистика `stats_v2` идёт по своему каналу `_persistStats` (не использует `cloudSet`), потому что у неё своя загрузка/слияние через `statsLoad` и она не вызывается до DOMContentLoaded.

Вне Telegram (`!inTelegram()`) `cloudSet` работает как обычный `localStorage.setItem` — приложение в браузере полностью функционально, просто без облачного бэкапа.

## Контракт хранилища статистики (stats_v2)

Единое хранилище для записи и чтения. Источник истины — Telegram CloudStorage когда приложение в Telegram, плюс localStorage всегда (offline-кэш и единственный backend в браузере).

**Ключ:** `stats_v2` (одинаковый в `localStorage` и в `Telegram.WebApp.CloudStorage`).

**Схема (JSON):**

```js
{
  version: 2,
  byDate: {
    "YYYY-MM-DD": {
      overall: { time, examples, mistakes, sessions,
                 examplesByOp: [+, −, ×, ÷],
                 mistakesByOp: [+, −, ×, ÷] },
      big:     { ... те же поля ... },
      small:   { ... те же поля ... }
    },
    ...
  }
}
```

- **`time`** — суммарное время по примерам (секунды).
- **`examples`** — количество решённых примеров.
- **`mistakes`** — количество примеров с хотя бы одной ошибкой.
- **`sessions`** — сколько раз пользователь нажимал «Начать».
- **`examplesByOp[4]`** / **`mistakesByOp[4]`** — разбивка по операциям `[+, −, ×, ÷]`.

**Public API (см. [main.js](main.js)):**

- `statsLoad()` → Promise. Загружает хранилище в кэш. В Telegram — async через `CloudStorage.getItem` + слияние с localStorage (`_mergeStatsObjects`, max-per-field). Иначе sync через localStorage. Идемпотентно. Merge нужен для устойчивости к failed cloud-writes: если запись в облако не дошла offline, локальный инкремент не теряется при следующем заходе.
- `statsRecordSessionStart(category)` — вызывается из [fromHomeToExample()](main.js). Инкрементирует `sessions` в `overall` + в `category` (если `'big'`/`'small'`).
- `statsRecordExample(category, opIndex, timeSec, hadMistake)` — вызывается из [keyboardClick('enter')](main.js) и [fromExampleToHome(1)](main.js). Инкрементирует `time`, `examples`, `mistakes`, `examplesByOp`, `mistakesByOp`.
- `statsByCategory(category)` → `{dateKey: entry}` для рендера страницы статистики.

**Категория раунда (`big` / `small` / `null`):** определяется через `localStorage.activePreset`. Ставится в `setActivePreset('big'|'small')` при клике на «Большой»/«Малый». Сбрасывается в `null` при любом ручном изменении чекбокса или ползунка (=> раунд учитывается только в `overall`).

**Миграция legacy:** старая схема `stats` (массив `[monthIdx, [t,e,m], ...]` за один месяц) автоматически конвертируется при первой загрузке через `_migrateLegacyArray()`. Данные кладутся только в `overall` (нет разбивки по операциям и категориям, sessions = 1 если в день что-то решали). После миграции хранилище пересохраняется под новым ключом.

**Ротация месяца:** больше не нужна как отдельная операция — формат byDate с ISO-датами хранит произвольное окно дней. Старые месяцы либо остаются в byDate, либо могут быть подчищены в будущем (см. TODO).

**Когда пишется:** на каждом правильном ответе в `keyboardClick('enter')`, на старте раунда в `fromHomeToExample()`, на нажатии «домой» в середине раунда — `fromExampleToHome(1)` (записывает текущий недорешённый пример как попытку).

**Debug:** `window.regenerateMockStats()` засеивает реалистичные данные на 35 дней в `stats_v2`, `window.clearMockStats()` обнуляет хранилище.

## Подсчёт ошибок (две переменные)

- **`mistake`** — флаг «была ли ошибка в текущем примере» (0 или 1). Передаётся в `statsRecordExample(..., hadMistake)` при фиксации примера, потом синхронно сбрасывается в `0` (раньше сброс жил внутри async CloudStorage-callback и работал только в Telegram — баг исправлен в рефакторинге stats_v2).
- **`totalMistake`** — счётчик ошибок за весь раунд, выводится в сообщении «Ошибки: N» после раунда.

`mistakeTwo` оставлен в коде, но реально дублирует `mistake` — раньше нужен был для отдельного канала записи, теперь его можно почистить.

Логика «1 пример = максимум 1 ошибка, сколько бы раз ты ни нажал Enter с неправильным ответом» обеспечивается тем, что `mistake = 1` (не `++`).

## Логика генерации примеров — [`setExample()`](main.js#L952)

1. Случайно выбирается операция из тех, у кого чекбокс включён (`values[0..3] == "true"`).
2. Числа генерируются в выбранном диапазоне:
   - Для **`+`** и **`−`** — из диапазона `values[5..6]`.
   - Для **`⋅`** и **`∶`** — из диапазона `values[7..8]`.
3. **Вычитание:** всегда переставляет операнды так, чтобы первое было больше — ответ неотрицательный.
4. **Деление:** генерирует пару `(a, b)`, потом подменяет первое число на `a*b`, чтобы деление было нацело. Запрещены 0 и 1 как операнды (см. [main.js:1006](main.js#L1006)).
5. **Уникальность примеров в раунде:** проверка против уже сгенерированных `examples` — не должны совпадать операция-и-ответ, не должны повторяться операнды. Логика проверки запутанная (флаг `a` и цикл с `exit = 100`); при рефакторинге упростить.
6. `examples` хранит плоский массив по 4 элемента на пример: `[op, n1, n2, answer, op, n1, n2, answer, ...]`.

**Известный баг:** в ветке вычитания на [main.js:989](main.js#L989) `if (numberOne = numberTwo)` — это присваивание, не сравнение. Нужно `==` или `===`. Не падает только потому, что внешний цикл всё равно выходит после первой удачной итерации.

## Три блока ползунков (rangeslider) — самый громоздкий код

Один и тот же шаблон повторён трижды с суффиксами:

| Назначение | CSS-классы | Min/Max | Дефолт |
|---|---|---|---|
| Диапазон для `+`/`−` | `.lower` / `.upper` / `.fill` / `.easy-basket-lower` / `.easy-basket-upper` | 0–300 | 0–100 |
| Диапазон для `⋅`/`∶` | `*-double` | 0–50 | 0–20 |
| Количество примеров | `*-three` | 5–25 | 10 |

«Three» — на самом деле одиночный ползунок (только нижний), но код написан как для двойного, и в HTML отсутствует элемент с классом `.upper-three`, хотя в JS он используется (`$('.upper-three').attr('max')` вернёт `undefined`). Это работает «случайно». **При рефакторинге эти три блока обязательно сводятся в один компонент.**

Функции `dinamicRange()`, `smallRange()`, `bigRange()` — это три почти идентичные копии кода обновления ползунков. ~600 строк дублирующейся логики.

## Пресеты «Малый»/«Большой» и спецпользователь

- **«Малый»** ставит все чекбоксы включёнными (включая таймер), диапазон +/− = 2–20, диапазон ⋅/∶ = 2–10.
- **«Большой»** ставит чекбоксы операций ВКЛ, таймер ВЫКЛ (`checkboxes[4].checked = false`), диапазоны больше.
- **Спецветка для `tg_username == 'alexander_drozd'`:** в `bigRange()` ему даются увеличенные диапазоны (+/− = 200–500, ⋅/∶ = 15–35). Это персональная настройка, заложенная в код. При рефакторинге решить: вынести в конфиг или убрать.

## Темы

- Один файл [thems/themes.css](thems/themes.css) хранит все темы как наборы CSS-переменных под селекторами `:root[data-theme="..."]`. Активная тема определяется атрибутом `data-theme` на `<html>` — JS просто меняет атрибут, ничего не подгружая. Этот же файл содержит и правила-привязки переменных к элементам (`background-color: var(--bg)` и т.д.), и фиксированные цвета превью-кружков `.theme-swatch-*`.
- Доступные темы: `light` (дефолт), `white`, `dark`, `black`, `blue`, `green`, `purple`, `orange`. 8 свотчей в HTML имеют атрибут `data-theme-name` — он же значение для `data-theme`. Порядок: нейтральные (от светлого к тёмному) → цветные.
- Один делегированный обработчик в `main.js` на `#conteiner` ловит клик по любому `.theme-swatch` и вызывает `applyTheme(name)`. `applyTheme` валидирует имя против массива `THEMES` (фоллбэк на `THEME_DEFAULT` если приходит мусор — нужно для миграции пользователей со старым `userTheme='standart'`), пишет в `localStorage.userTheme` и проставляет класс `.active-theme` на выбранный свотч (визуальное кольцо вокруг него через `outline`).
- Тень — одна переменная `--shadow` на тему (а не пара color+offset). Тонированная под цвет темы для цветных, нейтральная для нейтральных. Применяется на кнопки, `.example-answer`, `.graph-conteiner` через `box-shadow: var(--shadow)` в [thems/themes.css](thems/themes.css).
- На старте в `DOMContentLoaded` вызывается `applyTheme(localStorage.getItem('userTheme'))` — один раз, до этого момента браузер уже показал дефолтную светлую тему (она задана на `:root` без атрибута), так что FOUC минимальный.

## Таймер

- Кастомная реализация на `setInterval(startTimer, 10)`: переменные `tens` (сотые секунды) и `seconds`. Не использует `Date.now()` — будет дрейфовать при нагрузке/throttle вкладки.
- **Виден или скрыт** в зависимости от `values[4]` (чекбокс таймера). При выключенном таймере счётчик всё равно тикает в фоне — он нужен для статистики, просто HTML не обновляется.
- `TimeForSave` / `TimeForSaveOld` — для разбиения общего времени на куски «время на пример» (разница с предыдущей точкой). Это нужно для инкрементальной записи в `stats`.

## Графики

- Morris.Line на jQuery+Raphael. Три графика: примеры/время/ошибки по дням месяца.
- `addDivForGraph()` / `deleteDivForGraph()` пересоздают контейнеры при каждом открытии страницы — иначе Morris при повторных вызовах рендерил несколько слоёв hover.
- `graphToToday()` прокручивает контейнер по горизонтали так, чтобы текущий день оказался в центре экрана (использует `setTimeout(100)` для ожидания отрисовки — хрупко).
- График ошибок показывает **долю правильных ответов** `(examples - mistakes) / examples`, не абсолютное число ошибок (см. [main.js:178](main.js#L178)).

## Мобильная адаптация

- В viewport отключён зум: `user-scalable=no, maximum-scale=1.0`.
- Везде `vh`/`vw` — макет ломается на широких экранах (есть фиксированная `.conteiner { width: 1000px }`, выглядит как остаток отладки).
- `user-select: none` на всём — чтобы случайные тапы не выделяли текст.

## Что важно при реформировании

- Контракт `values` (порядок индексов) расползся по всему коду — менять надо синхронно или сначала ввести объект-обёртку.
- Контракт `stats_v2` централизован: запись только через `statsRecordSessionStart`/`statsRecordExample`, чтение через `loadStats`/`statsByCategory`. При расширении схемы (например, добавление поля) обновить `emptyCatDay()`, мигратор `_migrateLegacyArray`, и нормализатор `_normalizeStats`.
- jQuery нужен только для ползунков и Morris. Если убирать ползунки на нативный компонент или переписать — jQuery уйдёт целиком.
- Закомментированный блок в начале [main.js:9-16](main.js#L9-L16) — старый редирект для встраивания в CV-сайт автора, не используется.
